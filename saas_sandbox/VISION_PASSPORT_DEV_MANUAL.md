**技術撰寫**：Antigravity (CTO)

## 🚀 壹、 核心開發大平臺思維 (AI-Driven Development)

傳統的軟體開發模式（由工程師從零手刻 HTML/CSS/JS 與後端）已經不符合本集團的敏捷戰略。
本 APP 將全面採用 **AI 原生工作流 (AI-Native Workflow)**，最大化開發速度與介面質感。

### 技術堆疊矩陣 (The Tech Stack)

1. **前端視覺與互動 (UI/UX) ➔ 【Stitch AI / v0 類工具】**
   *   **定位：** 我們的專屬 UI 設計師與前端工程師。
   *   **任務：** 負責將大叔腦海中的「玻璃擬物化 (Glassmorphism)」、「全家帳號無縫切換」、「手機滑動轉場」等概念，快速生成高質感的 React/Vite 或純前端程式碼。
   *   **原則：** 以「Mobile-First (手機好用為大原則)」為首要前提。
2. **後端大腦與邏輯 (Backend & Intelligence) ➔ 【Google AI Studio (Gemini Pro)】**
   *   **定位：** 我們的首席驗光分析師與雲端大腦。
   *   **任務：** 接收前端傳來的純生硬數據（如 SPH -2.50, AXL 24.2），透過 Gemini 模型將其轉化為「溫暖、具備商業導購能力」的白話文解釋。
   *   **優勢：** 取代傳統寫死的 `if-else` 邏輯，擁有能讀懂歷史趨勢並給出具體建議的超強大腦。
3. **終端載體 (Client) ➔ 【LINE LIFF / PWA】**
   *   **定位：** 使用者觸點。
   *   **任務：** 將 Stitch 生成的 UI 裝載進 LINE 裡，免除下載 APP 的阻力。

## 🛠️ 貳、 開發實作 SOP (Standard Operating Procedure)

### Phase 1: 詠唱 UI/UX (透過 Stitch AI)
**執行者：CTO 撰寫 Prompt 餵給 UI AI**

*   **Prompt 範例 (給 Stitch AI 的指令)：**
    > 「我需要一個以 Mobile-First 設計的『視力健康數位護照』Web App。首頁要有家庭成員的卡片清單（如：媽媽、大寶）。點擊卡片後，擁有像原生 App 一樣的滑入轉場，進入個人詳細頁面。
    > 
    > **風格要求：** 科技醫療感、暗黑背景搭配霓虹藍/紫色漸層光暈 (Glassmorphism 玻璃質感)。
    > **核心元件：** 需要一個『歷年視力成長折線圖』的預留區塊，以及一個『AI 處方建議』的卡片。請使用 Tailwind CSS 幫我生成高品質的 UI。」

### Phase 2: 搭建智慧 API 閘道 (透過 Google AI Studio)
**執行者：CTO 串接 Gemini API**

*   **實作流程**：
    1.  前往 Google AI Studio 取得 API Key。
    2.  定義 `System Prompt` (賦予 AI 靈魂) ➔ 「你是一位名為『目鏡大叔』的資深台灣驗光師，擅長用接地氣且溫暖的口吻向家長解釋視力數據...」
    3.  建立一套 API 端點 (例如透過 Vercel Serverless Functions 或本地 Python API)。當使用者在前端 APP 點擊「大寶的護照」時，系統將大寶的歷史數據 JSON 丟給 Google AI Studio。
    4.  Google AI Studio 回傳格式化的 HTML/Markdown 建議，前端將其渲染在畫面上。

### Phase 3: 黑盒串接與去識別化資料庫
*   APP 登入時取得 LINE UserID，但不紀錄真實姓名（僅紀錄代號如：User_A_Kid_1）。
*   每次從 Google AI Studio 回傳的分析紀錄，同步一份 (僅存數值與標籤) 至 Supabase 或 Firebase，作為未來的「大數據研究池」。

## 🎯 參、 應用場景預覽 (The User Journey)

1. **情境**：媽媽帶大寶在驗光所剛做完半年一度的檢查。
2. **觸發**：媽媽的手機 LINE 收到推播：「大寶的最新視力報告已出爐！點此查看大叔的分析。」
3. **體驗 (Stitch AI 的功勞)**：媽媽點擊連結，畫面絲滑地展開，呈現極具質感的深色玻璃卡片 UI。
4. **感動 (Google AI Studio 的功勞)**：媽媽看到卡片上寫著：
   > 「*媽媽您好，大叔看了大寶這次的數據。這半年來眼軸長度控制得非常完美，完全沒有增加！您可以放心了，這代表我們上次更換的『周邊離焦鏡片』發揮了極大作用。請繼續保持每天兩小時的戶外活動喔！*」
5. **商業價值**：媽媽瞬間覺得這家驗光所極度專業且貼心，立刻在群組向其他媽媽推薦本診所。

## 📱 肆、 UX 首要原則：漸進式揭露 (Progressive Disclosure)

> CEO 定調：**「最基本的紀錄可以運作，培養出習慣之後才會完整填寫。不要先強制。」**

### 三層漸進式填寫設計

| 層級 | 使用者心態 | 需要填的 | APP 回饋 |
| :--- | :--- | :--- | :--- |
| **Lv.1 隨手記** | 「先記就好」 | 日期 + 左右眼度數 (30 秒搞定) | ✅ 折線圖出現第一個點 |
| **Lv.2 開始在意** | 「怎麼又增加了」 | 補填散光、眼軸、控制方案 | 📈 AI 建議變得更精準 |
| **Lv.3 深度關心** | 「全家人都要顧好」 | 健康狀況勾選、行為打卡 | 🏆 解鎖「完整健康報告」 |

### 給 Stitch AI 的 UI 規則：
*   首頁新增紀錄時，**只顯示 Lv.1 欄位**（日期 + 度數）。
*   進階欄位藏在「想填寫更多？」的摺疊按鈕裡。
*   當 AI 偵測到度數異常時，主動引導：「如果能補填眼軸長度，大叔就能做更精準的分析喔！」→ 用戶自然主動升級至 Lv.2。

## 🔄 伍、 三階段資料輸入策略 (Cold-Start Engine)

初期並非所有驗光所都有我們的 SaaS 系統，因此必須設計多重資料來源：

| 優先度 | 資料來源 | 家長操作 | 體驗等級 |
| :--- | :--- | :--- | :--- |
| ⭐⭐⭐ | **SaaS 自動同步** | 什麼都不用做 | 頂級 (零操作) |
| ⭐⭐ | **📸 拍照 OCR 辨識** | 拍一張處方箋照片 | 很方便 (拍一張照) |
| ⭐ | **白話文手動輸入** | 回答「右眼幾度？左眼幾度？」 | 堪用 (打兩個數字) |

### OCR 模式流程 (Gemini Vision)：
1. 家長按「📷 新增驗光紀錄」
2. 拍下紙本處方箋或螢幕截圖
3. Google AI Studio (Gemini Vision) 自動辨識 SPH/CYL/AXIS
4. 系統確認：「右眼近視 250 度，左眼 225 度，對嗎？」
5. 家長按確認 → 存入

### 冷啟動的「陽謀」：
*   **初期**：靠 OCR + 手動輸入 → 先衝 C 端用戶量
*   **中期**：家長反問驗光所「你能不能傳到我的視力存摺？」→ 反向推動 B 端訂閱 SaaS
*   **後期**：SaaS 全自動同步 → 完美閉環

## 💡 CTO 結語
本手冊定義了 Vision Passport 的三大支柱：
1. **AI 雙引擎**：Stitch 畫皮、Google 鑄魂。
2. **漸進式 UX**：30 秒能用，30 天養習慣，30 週收數據。
3. **冷啟動引擎**：C 端先行，反向逼 B 端訂閱，最終生態閉環。
